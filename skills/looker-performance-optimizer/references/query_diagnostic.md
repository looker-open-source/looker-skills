# Query Diagnostics & execution Plan Manual

This reference handbook details the standard operating procedures for executing performance diagnostics, analyzing physical SQL, and interpreting database execution plans (**EXPLAIN Plans**) to locate and resolve database bottlenecks.

Server-side LookML compilation does not check query performance. Developers and AI agents **MUST** follow this diagnostic manual to identify inefficiencies in the semantic and physical database layers.

---

## 1. The Parallel Diagnostics Workflow (Subagent-Assisted)

To optimize token efficiency and prevent conversation bloat, the main agent delegates query diagnostics to specialized subagents in parallel.

```
[Main Agent] ➔ [Spawn SQL Profiler Subagents] ➔ [Subagents run MCP query] ➔ [Run EXPLAIN plan] ➔ [Analyze Bottleneck] ➔ [Return Pruned 1-Paragraph Recommendation] ➔ [Main Agent coordinates LookML Refactoring]
```

### The Subagent Diagnostics Protocol:
1.  **Retrieve SQL**: The subagent calls the Looker MCP tool `query` or `query_sql` to fetch the exact physical SQL statement generated by the slow dashboard tile.
2.  **Execute EXPLAIN**: The subagent runs the SQL statement prefixed with `EXPLAIN` (or the dialect's profiling prefix) against the database connection.
3.  **Perform Schema & Key Audit**: The subagent calls `get_connection_table_columns` to check table sizes, indexes, and partition keys.
4.  **Synthesize Recommendations**: The subagent analyzes the execution plan, identifies the bottleneck (e.g., table scan, Symmetric Aggregations, disk spill), compiles a condensed report, and reports back to the main agent.

## 2. The Symmetric Aggregations Diagnostic Protocol

Looker automatically triggers **Symmetric Aggregations** when a query calculates a measure (like `SUM` or `COUNT`) across a join that has a one-to-many relationship. Looker does this to prevent double-counting (data fan-out).

> [!NOTE]
> **The Cost of Hashing**: Looker implements Symmetric Aggregations by wrapping values in complex hashing functions (e.g., `SUM(DISTINCT (CAST(COALESCE(users.age,0) AS NUMERIC) + (CAST(CAST(CONCAT('0x', ...))))))`). This forces the database to compute SHA1 hashes, string concatenations, and primary-key groupings on every single row before aggregating, resulting in massive CPU and memory consumption.

### The Uniqueness Audit Algorithm:
If a query contains `_distinct` aggregate functions in its SQL (indicating Symmetric Aggregations are active), the agent **MUST** audit whether the join relationship is configured correctly.

1.  **Identify the Join Key**: Locate the join condition in the Explore LookML (e.g., `join: users { sql_on: ${orders.user_id} = ${users.id} ;; relationship: many_to_one }`).
2.  **Run the Uniqueness Test**: Execute this SQL query against the target database connection to verify if the join key of the "one" side (e.g., `users.id`) is truly unique:
    ```sql
    SELECT users_id, COUNT(*) 
    FROM (
      -- Replace with the actual table and column name
      SELECT id AS users_id FROM dataset_name.users
    ) 
    GROUP BY 1 
    HAVING COUNT(*) > 1 
    LIMIT 5;
    ```
3.  **Analyze the Test Results**:
    *   **Case A: Query returns 0 rows (Key is Unique)**: The join key is a true primary key. Looker is triggering Symmetric Aggregations **unnecessarily** because the developer misconfigured the LookML `relationship` parameter.
        *   *Remediation*: Change the join relationship in LookML to `many_to_one` or `one_to_one`. Looker will immediately compile clean, native SQL aggregates (e.g., `SUM(col)` instead of `SUM(DISTINCT ...)`), saving massive database CPU.
    *   **Case B: Query returns rows (Key is NOT Unique)**: The join key has duplicate records. Looker is correctly protecting data integrity from a Cartesian fan-out.
        *   *Remediation*: Do not change the relationship. Instead, pre-aggregate the "many" side of the table into a **Persistent Derived Table (PDT)** first, then join the aggregated PDT to the primary explore on a clean `one_to_one` or `many_to_one` unique basis.

## 3. SQL Explain Plan Cheat Sheet (Dialect-Agnostic)

Run the generated SQL prefixed with `EXPLAIN` (or dialect-equivalent profiling) in the database console. Audit the execution plan using the following cheat sheet to locate physical bottlenecks:

### A. Google Cloud BigQuery
*   **How to Run**: Prefix the query with `EXPLAIN` or view the Execution Graph in the console.
*   **Key Bottlenecks to Audit**:
    *   `TABLE_SCAN` without pruning: The query is scanning the entire physical table.
        *   *Fix*: Add partition filters (date ranges) in the Explore/Dashboard filters, or cluster the PDT on the filtered fields.
    *   `BYTES_SPILLED_TO_DISK` / `LIMIT_EXCEEDED`: The intermediate join or sort was too massive to fit in BigQuery's RAM, spilling to local SSD or failing.
        *   *Fix*: Reduce the number of columns selected, optimize join order, or pre-aggregate in a PDT.
    *   High `SLOT_MILLIS`: Excessive CPU consumption.
        *   *Fix*: Eliminate Symmetric Aggregations or replace complex string regex with pre-computed fields.

### B. Google Cloud Spanner (GoogleSQL)
*   **How to Run**: Prefix the query with `EXPLAIN` or use Spanner Query Statistics to extract the execution tree.
*   **Key Bottlenecks to Audit**:
    *   `Distributed Cross Apply` / `Nested Loop`: Spanner is executing a nested loop, querying table B once for every row in table A. Highly inefficient for large datasets.
        *   *Fix*: Ensure join columns have **identical data types** (e.g., do not join an `INT64` key to a `STRING` key, which forces a type coercion scan).
    *   `Scan` without Index: Spanner is scanning the entire table on disk.
        *   *Fix*: Ensure the query filters match the table's primary keys or secondary indexes.
    *   `Hash Join`: Used for large, unsorted joins.
        *   *Fix*: Build a PDT clustered on the join keys, or pre-sort/pre-aggregate the datasets.

### C. Snowflake
*   **How to Run**: View the Query Profile tab in the Snowflake UI.
*   **Key Bottlenecks to Audit**:
    *   `Partitions scanned: 100%`: Snowflake is performing a full table scan, failing to apply partition pruning.
        *   *Fix*: Ensure explore filters align with the table's clustering keys.
    *   `Spill to Local Storage` / `Spill to Remote Storage`: The virtual warehouse ran out of memory, writing temp files to disk.
        *   *Fix*: Scale up the virtual warehouse size (e.g., from X-Small to Medium) temporarily during PDT rebuilds, or pre-aggregate the joins.
    *   `Query result reuse: 0%`: Snowflake is rebuilding the query instead of utilizing the metadata cache.
        *   *Fix*: Align caching datagroups to prevent unnecessary cache invalidation.

## 4. Telemetry & Diagnostics Checklist
- [ ] Spawn a specialized subagent for every slow dashboard query to run parallel profiling.
- [ ] Intercept the generated SQL and identify if Symmetric Aggregations (`_distinct`) are active.
- [ ] Run the **Uniqueness Audit SQL** to verify if the join key is unique on the "one" side of the join.
- [ ] Run an `EXPLAIN` plan on the generated SQL against the target database connection.
- [ ] Inspect the execution plan for table scans, nested loops, and disk spills.
- [ ] Check column data types on join keys using `get_connection_table_columns` to ensure they are identical.
- [ ] Synthesize all subagent findings into a condensed, 1-paragraph performance recommendation.
- [ ] Apply all refactorings surgically in-place to preserve developer comments and protect unrelated code.
